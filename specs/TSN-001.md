TSN-001: Migration post-quantique ML-DSA-65 → SLH-DSA  
Status: Draft  
Author: Laila.H  
Created: 2024-12-08  
Updated: 2024-12-08

## Abstract
Cette spec définit le plan de migration de la signature post-quantique de Trust Stack Network depuis ML-DSA-65 (FIPS 204) vers SLH-SHA-128s (FIPS 205) afin de renforcer la résistance à la cryptanalyse quantique tout en conservant la compatibilité descendante pendant 2 époques (~14 jours).

## Motivation
ML-DSA-65 (Module-Lattice-Based Digital Signature Algorithm) offre une sécurité post-quantique de NIST-Level 2, mais repose sur la difficulté des problèmes en treillis. SLH-SHA-128s (Stateful Hash-Based Signatures) fournit une sécurité informationnelle (basée uniquement sur la résistance collisionnelle de SHA-256) et est recommandé par le NIST pour les applications à haute assurance.  
Objectifs :  
1. Renforcer la sécurité à long terme contre les attaques quantiques.  
2. Réduire la surface d'attaque cryptographique à un seul primitif de hachage (SHA-256).  
3. Maintenir la compatibilité descendante pour les nœuds non mis à jour.

## Specification

### 1. Paramètres SLH-SHA-128s
| Paramètre | Valeur | Unité |
|-----------|--------|-------|
| n         | 32     | bytes |
| w         | 16     | –     |
| h         | 66     | –     |
| d         | 22     | –     |
| k         | 33     | –     |
| sig_len   | 17 088 | bytes |
| pk_len    | 64     | bytes |
| sk_len    | 128    | bytes |

### 2. Structure des clés
```rust
pub struct SlhDsaKeypair {
    pub pk: [u8; 64],     // SLH public key
    pub sk: [u8; 128],    // SLH secret key
    pub idx: u64,         // Compteur d'état (stateful)
}

### 3. Format des transactions de migration
Une transaction `MigrateSignatureScheme` est créée par le wallet et contient :  
- `old_pk` : clé publique ML-DSA-65 (32 B)  
- `new_pk` : clé publique SLH-SHA-128s (64 B)  
- `sig_old` : signature ML-DSA-65 de `hash(old_pk || new_pk)`  
- `sig_new` : signature SLH de `hash(old_pk || new_pk)` (pour preuve de possession)  
- `epoch` : numéro d'époque de soumission

### 4. Cycle de vie
stateDiagram-v2
    [*] --> ActiveMLDSA: Époque N
    ActiveMLDSA --> Pending: Migration soumise
    Pending --> DualSig: Époque N+1
    DualSig --> SLHOnly: Époque N+2
    SLHOnly --> [*]: Migration complète

### 5. Règles de validation (consensus)
1. Durant `DualSig`, les blocs acceptent :  
   - Transactions signées ML-DSA-65 OU SLH-SHA-128s.  
   - Pas de mélange des deux dans une même tx.  
2. Après `SLHOnly`, seules les signatures SLH sont valides.  
3. Le compteur `idx` SLH doit être strictement croissant par compte.  
4. Une seule transaction de migration autorisée par compte.  
5. Frais de migration fixes : 0.1 TSN brûlés.

### 6. Algorithme de vérification côté nœud
fn verify_migration(tx: &TxMigration) -> Result<()> {
    // 1. Vérifier la signature ML-DSA-65
    let msg = hash(&tx.old_pk, &tx.new_pk);
    ml_dsa_verify(&tx.old_pk, &msg, &tx.sig_old)?;

    // 2. Vérifier la preuve de possession SLH
    slh_verify(&tx.new_pk, &msg, &tx.sig_new)?;

    // 3. Vérifier l'époque
    ensure!(tx.epoch >= current_epoch() - 1, "Too late");
    ensure!(tx.epoch <= current_epoch() + 1, "Too early");

    // 4. Vérifier le compteur d'état
    let idx = fetch_slh_idx(&tx.new_pk)?;
    ensure!(idx == 0, "Already migrated");
    Ok(())
}

### 8. Scripts de migration automatique
Un binaire `tsn-migrate` fourni :  
- Génère une nouvelle paire SLH.  
- Crée et soumet la transaction de migration.  
- Attend 2 époques et supprime l'ancien keystore ML-DSA.  

Utilisation :  
tsn-migrate --wallet ~/.tsn/wallet.json --fee 0.1 --confirm

## Rationale
- **Pourquoi SLH-SHA-128s et pas SLH-SHA-192s ?**  
  128s suffit à NIST-Level 1 (≈ 160 bits de sécurité classique) et garde une taille de signature raisonnable (17 kB vs 50 kB).  
- **Pourquoi un compteur d'état externe ?**  
  Le noyau SLH est stateful ; TSN conserve le compteur dans `state.rs` pour garantir l'unicité.  
- **Pourquoi 2 époques de fenêtre ?**  
  Permet aux nœuds hors ligne de se synchroniser sans briser la chaîne ; inspiré de Sapling → Orchard chez Zcash.

## Security Considerations
- **Clés ML-DSA compromises après migration**  
  Même si une clé ML-DSA devient faible, l'utilisateur a déjà migré ; les fonds restent protégés par SLH.  
- **Réutilisation d'index SLH**  
  Le consensus rejette strictement toute tx avec `idx ≤ last_used`; une attaque de réutilisation est impossible.  
- **Taille de signature**  
  17 kB par signature augmente la taille des blocs d'environ 8 % ; le paramètre `MAX_BLOCK_SIGS` passe de 2048 à 1920.  
- **Randomisation du nonce**  
  Le RNG de SLH est nourri par `ChaCha20Rng` avec graine dérivée de `OsRng + block_hash` pour éviter la faute de rejeu.

## Backward Compatibility
- Les anciens wallets continuent de fonctionner jusqu'à l'époque `N+2`.  
- Un champ `legacy_support_until` est inclus dans l'en-tête de bloc pour signaler aux clients légers la fin de la compatibilité.  
- RPC : `get_migration_status` retourne l'état actuel (`ActiveMLDSA`, `DualSig`, `SLHOnly`).
