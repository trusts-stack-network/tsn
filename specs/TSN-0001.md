TSN-0001: Migration ML-DSA-65 → SLH-DSA
Status: Draft  
Author: Laila.H  
Created: 2024-12-19  
Updated: 2024-12-19  

## Abstract
Ce TSN définit le plan de migration de la signature post-quantique de Trust Stack Network depuis ML-DSA-65 (FIPS 204) vers SLH-DSA (FIPS 205) afin d’éliminer la dépendance à des hypothèses supplémentaires (GR.3.2) et renforcer la sécurité à long terme. Il spécifie les paramètres cryptographiques, les vecteurs de test, le format de transaction étendu et le calendrier de déploiement.

## Motivation
1. ML-DSA-65 repose sur la difficulté des réseaux euclidiens + Module-LWE/Module-SIS ; SLH-DSA repose uniquement sur des fonctions de hachage sécuritaires (SHA-2/SHAKE).  
2. Les récents progrès en cryptanalyse des réseaux (cf. [NTU24]) rendent prudents l’usage exclusif de signatures à base de réseaux.  
3. SLH-DSA offre une marge de sécurité plus conservatrice (niveau 5, 256 bits de sécurité classique & post-quantique) avec seulement des hypothèses standard sur les hachages.  
4. Une migration coordonnée préserve la compatibilité descendante et évite une rupture de chaîne.

## Specification

### 1. Paramètres SLH-DSA retenus
| Paramètre | Valeur | Justification |
|---|---|---|
| Instance | SLH-DSA-SHAKE-256f | « f » = fast, 17 088 B de clé publique, 49 B de signature |
| n | 32 | longueur des sorties de hachage / adresse |
| h | 66 | hauteur hypertree |
| d | 22 | nombre de niveaux dans l’hypertree |
| k | 33 | paramètre FORS |
| t | 68 | paramètre FORS |
| w | 16 | paramètre WOTS+ (w = 16 ⇒ log = 4 bits par chaîne) |
| len | 67 | nombre de chaînes WOTS+ (len = len₁ + len₂) |

### 2. Format des clés
```rust
// src/crypto/pq/slh_dsa.rs
pub const SLH_PK_LEN: usize = 1_376; // 32 + 32*17 (cf. FIPS 205, tab. 3)
pub const SLH_SK_LEN: usize =  64;  // graine secrète + graine publique
pub const SLH_SIG_LEN: usize = 2_436;

### 3. Extensions de transaction
On ajoute un champ `signature_type` (1 byte) dans `Transaction`:
0x00 = ML-DSA-65 (legacy)
0x01 = SLH-DSA-SHAKE-256f
La serialisation binaire reste inchangée : champ optionnel TLV (tag 0xFA, length 1).

### 4. Adresses et portefeuille
Les `Address` actuelles (32 bytes) restent valides ; on dérive simplement la clé publique SLH via:
pk_slh = H(pk_seed || address_index)
où `address_index` est un compteur 64 bits stocké dans l’`Account`.

### 5. Calendrier de migration
| Phase | Blocs | Commentaire |
|---|---|---|
| 0 | 0 – 720_000 | ML-DSA-65 uniquement |
| 1 | 720_001 – 750_000 | SLH autorisé, pas encore mandatory |
| 2 | ≥ 750_001 | SLH requis ; ML-DSA-65 rejeté (soft-fork) |

## Rationale
- Pourquoi SLH-256f et non 128s ? 256f offre la meilleure taille de signature parmi les instances niveau 5 ; 128s réduit la taille de clé mais reste niveau 3.  
- Pourquoi pas une hybridation (ML-DSA + SLH) ? Double la taille de signature (> 4 kB) et complexifie la vérification ; nous préférons un switch net.  
- Pourquoi un champ `signature_type` plutôt que deux champs ? Un seul byte minimise la bloat et reste forward-compatible (types 0x02-0xFF réservés).

## Security Considerations
1. Hypothèses : on suppose que SHA-3/SHAKE-256 restent préimages & collision-résistantes.  
2. Randomness : le wallet doit utiliser un RNG conforme à NIST SP 800-90A rev. 1.  
3. Rejeu : la `RotateKey` transaction inclut un nonce 64 bits et un compteur de compte pour éviter la ré-utilisation.  
4. Compatibilité descendante : les nœuds en phase 0 rejettent silencieusement les blocs contenant des signatures SLH ; un bit de version `BLOCK_V3` est levé dès la phase 2.

## Backward Compatibility
- Les anciennes transactions ML-DSA-65 restent valides jusqu’au bloc 750 000.  
- Les nœuds non mis à jour refusent la chaîne après le bloc 750 001 (soft-fork).  
- Le wallet détecte automatiquement la phase via `consensus::MIGRATION_HEIGHT` et propose la migration en conséquence.
