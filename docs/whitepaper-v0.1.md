## Abstract
Trust Stack Network (TSN) is une blockkchain of theyer 1 post-quantum designed for confidentiality native, la scalability modulaire and la resistance cryptographic to long terme.  
Elle combine of the post-quantum signatures (SLH-DSA), of the ZK proofs-STARK (Halo2) and un hash internal resistant to la cryptanalysis quantique (Poseidon2) pour offrir of the transferts confidentiels and un consensus decentralized without compromettre la security future.

---

## Introduction
Les blockkchains actuelles souffrent de deux limites majeures :
1. The post-quantum randomness : les schemes to public key classiques (ECDSA, Ed25519, BN254) are vulnerables to algorithmes de Shor.
2. Le dilemme confidentiality/auditability : les mechanisms de confidentiality (zk-SNARKs) reposent souvent sur of the parameters d’installation (trusted setup) or of the courbes non post-quantums.

TSN propose une architecture qui :
- remplace fully the signatures to public key par of the constructions NIST-FIPS-204 (ML-DSA) and FIPS-205 (SLH-DSA) ;
- utilise of the ZK proofs-STARK (Halo2) without setup trust ;
- conserve une logique de state-transition verireliable par of the node lights thanks to of the commitments Merkle to hash quantique (Poseidon2).

## Transaction Model

### UTXO vs Account
TSN adopte le model **MIK (Merkleized Invoice Key-set)** : une UTXO enrichie d’un masque `r` and d’un `nullifier_key` derived en zero-knowledge.  
Beforeage : pas de adresse reusable en clair, pas de lien output-consommateur without secret key.

### Lifecycle
sequenceDiagram
    participant U as Wallet
    participant N as Network
    participant M as Mempool
    participant B as Block Producer
    U->>U: GenNote(amt, r)
    U->>U: ComputeCommit()
    U->>N: PushTx(tx, π halo2)
    N->>M: validateπ, checkNullifier
    M->>B: Propose
    B->>B: UpdateState(commit, nullifier)

| `memo` | 512 B | payload arbitrary |

## Consensus Layer

### PoQW – Proof of Quantum Work
- **Fonction :** `PoQW(B_header) = H_poseidon2(B_header || nonce)` with `H_output < target`
- **Verification :** node lights verify le STARK generated by the mineur prouvant le bon calcul of the hash without reveal `nonce`.
- **STARK circuit :** 12 000 constraints, proof ≈ 80 kB, verification ≈ 6 ms (x86).

### Difficulty ajustement
- Window : 600 blockks
- Algo : `new_diff = old_diff * (600 * T_target) / (T_observed)` with limite 4×/÷4 par window.

### Rewards & fees
- Halving every 210,000 blockks (Bitcoin-style)
- 92% reward to miner, 5% dev fees to treasury, 3% relay pool
- NO PREMINE — treasury accumulates through mining only

## State & Storage

### Arbre Merkle (Poseidon2, arity 16)
- **depth :** 32 → 2²⁶ feuilles max
- **leaf :** commitment 32 B d’une Note
- **root stored :** in `blockk.header.commit_root`

### Database
- Sled (KV) : colonnes `blockks`, `txs`, `notes`, `nullifiers`, `headers`
- Compactness automatic all les 24 h or >10 GB de delta

### Snapshots
- Each 10 000 blockks
- Format : zstd + bincoded `StateSnapshot { merkle_root, utxo_checksum, treasury_balance }`
- Verification : `SnapshotProof` Halo2 (TSN-0005)

## Security Considerations

### Surface d’attack post-quantum
- Signature : SLH-DSA & ML-DSA – security ≥ NIST-5 (256 bits post-quantum)
- Hash : Poseidon2 – structure sponge, non algebraic friendly pour Grover
- Forward secrecy : keys unshot par transaction (drift BIP-44 post-quantum)

### DoS Countermeasures
- Size maximale d’a blockkk : 2 MB
- Size maximale d’une proof Halo2 : 150 kB
- Prelim : 2 000 tx/s par pair before clearance mempool

### Audit & Formalization
- Specifications writtenes en Lean4 (directory `formal/`)
- Proofs de security : ROM+ QROM pour SLH-DSA, AGM pour Halo
